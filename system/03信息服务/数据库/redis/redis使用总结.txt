因其特性是运行在内存中的。好处是，速度非常快，处理海量数据时也相当快。但问题也就显现了：

1. 在 redis 进程重启后，总会有一些数据丢失，这取决于 dump 数据到硬盘的间隔时间。

2. 内存消耗非常大，如果还要 dump 数据，会产生新的 redis 进程，内存消耗可以大概按翻倍计算。

3. vm 模式可以节省约三分之二的内存，但 CPU 负载极高，高并发下不能及时响应请求。
   而且， redis 进程只运行在一个 CPU 或者核心上，不能通过增加 CPU 数量提升计算性能。

4. 如需备份，最好做主从模式在 slave 上备份，主服务器就可以关闭 dump ，以最快的速度运行。

5. redis 的主从切换较为简单，在线执行切换命令即可， slave 会重新加载本机的 dump 数据然后以 master 模式运行。
   但这也造成主从切换时有一段时间的服务中断，这视数据量大小时间有所不同，千万条数据大概要十分钟左右。

6. dump 出来的文件大概只有消耗内存的三分之一，备份文件大小不算太大，硬盘空间足够存放。
